1. Overview #

v2t turns raw insurance customer calls — or pre-labeled transcripts — into structured intelligence: extracted questions with intent and language metadata, semantic clusters of recurring concerns, canonical FAQs distilled from each cluster, and a memory graph of how concerns relate. A Next.js dashboard sits on top.

What it does

  • Ingests audio or pre-labeled transcripts via POST /ingest.
  • Transcribes audio with Sarvam.ai (Hindi + 10 Indic languages).
  • Extracts structured questions via Groq (openai/gpt-oss-120b, OpenAI-compatible).
  • Embeds with multilingual-e5-large (1024-dim) into pgvector.
  • Clusters with HDBSCAN; emits canonical FAQs and memory-graph edges.

Where it runs

  • Local Docker stack — 9 services (8 long-running + 1 init).
  • Postgres 16 + pgvector 0.8.2 + pgcrypto.
  • Redis 7 (Celery broker / result / cache).
  • MinIO (object storage for transcripts & artifacts).
  • Outbound: Groq HTTPS, Sarvam HTTPS, LangSmith HTTPS — all through the corporate MITM.

2. Architecture #

Live services in solid fill, skipped services in hatched fill. Arrows show request / data direction.

Data flow

  1. POST /ingest accepts an audio URL or a pre-labeled transcript JSON.
  2. Audio path: chunked on silence (30 s Sarvam limit) → Sarvam transcription → speaker heuristic → stitched transcript.
  3. Transcript → LLM extraction (Groq) → ExtractedQuestionORM rows.
  4. Each question → e5-large embedding → pgvector row. stalled today: GPU worker offline
  5. Periodic recluster (HDBSCAN) → SemanticCluster rows.
  6. Cluster → canonical FAQ + memory-edge inference.
  7. Dashboard reads via the 12 FastAPI routes.

6. Corporate TLS bundle #

The dominant source of build/runtime failures. Without trusting Cisco Umbrella + BajajLife roots, nothing HTTPS works inside containers.

Verify Groq from inside a container

docker exec v2t-api curl -sS -o /dev/null -w "HTTP %{http_code}\n" \
  -H "Authorization: Bearer $LLM_API_KEY" \
  https://api.groq.com/openai/v1/models
# expected: HTTP 200
Why the frontend uses node:20-slim instead of node:20-alpine

After TLS is trusted, Cisco Umbrella still corrupts the binary .apk payloads from dl-cdn.alpinelinux.org — every package fails signature verification (file format is invalid or inconsistent). Debian's deb.debian.org mirror serves over plain HTTP and passes through cleanly, so we swap to a Debian-based Node image.

If new corporate CAs ship

.\scripts\install-corp-ca.ps1
docker compose build api worker-cpu beat frontend
docker compose up -d --force-recreate api worker-cpu beat frontend

docker compose build api worker-cpu beat

docker cp ./app/api/routes/analytics.py v2t-api:/app/app/api/routes/analytics.py

9. Development log #

What changed since the initial commit (252386d), why, and where. Coloured dots: config / bug / fix.

9.1 .env created

phase: setup · files: .env
Compose was warning on every command because .env.example was the only env file present. Created .env from the template, then layered Bajaj-specific values: compliance_user / compliance_db, Groq, LangSmith, Firebase. Added OLLAMA_MODEL placeholder so the legacy ollama-init service's interpolation stops warning even though Ollama isn't started.

9.2 Corporate TLS bundle wired into every image

phase: infra · files: infra/certs/bajaj-root.pem, all 5 Dockerfiles, scripts/install-corp-ca.ps1
Exported Cisco Umbrella Root CA + BajajLife-Root-CA from the Windows root store. Each Dockerfile copies the PEM into /usr/local/share/ca-certificates/, runs update-ca-certificates, and sets REQUESTS_CA_BUNDLE / SSL_CERT_FILE / CURL_CA_BUNDLE / PIP_CERT / NODE_EXTRA_CA_CERTS. Switched the frontend from node:20-alpine to node:20-slim because the proxy corrupts binary .apk payloads even after TLS is trusted.

9.3 Build dependency pins

phase: deps · files: requirements.txt, frontend/components/CytoscapeGraph.tsx, frontend/types/modules.d.ts
  • indic-transliteration==2.3.66 did not exist on PyPI; bumped to 2.3.67.
  • Added ambient declare module "react-cytoscapejs"; shim.
  • cytoscape.Stylesheet[] was renamed in newer @types/cytoscape — loosened the typing instead of rewriting every CSS rule.
  • Frontend runner stage adds mkdir -p public because the repo has no frontend/public dir and Next.js standalone expects it.

9.4 FastAPI middleware ordering

phase: code · files: app/api/main.py
FastAPIInstrumentor.instrument_app(app) was called inside _lifespan. Recent Starlette refuses to add middleware after startup: 
RuntimeError: Cannot add middleware after an application has started
Moved OTel + logging init into create_app() before any add_middleware call.

9.5 Healthchecks

phase: infra · files: Dockerfile.beat, docker-compose.yml
Added procps to the beat runtime image so pgrep -f 'celery.*beat' works. Replaced the frontend's wget-based healthcheck with a Node-native http.get check (slim image has no wget).

9.6 Port remap

phase: infra · files: docker-compose.yml
Frontend remapped from host port 3000 → 3001; another node.exe on the host was already bound to 3000. Container internal port stays 3000.

9.7 Frontend → API proxy mis-pointed

phase: bug · files: frontend/next.config.js, frontend/Dockerfile
Two bugs in one chain:
  1. next.config.js defaulted to http://api:8000. The API listens on :8080.
  2. Next.js bakes rewrite destinations at build time, so the environment: block in compose never reached the build.
Fixed the fallback to http://api:8080 and declared ARG NEXT_PUBLIC_API_BASE_URL=http://api:8080 in the frontend Dockerfile so future overrides flow through.

9.8 Six API routes imported model classes under the wrong name

phase: bug · files: analytics.py, calls.py, search.py, feedback.py, faq.py, memory_graph.py
The ORM module exports ExtractedQuestionORM / ClusterMemberORM / etc., but six route files imported them under the non-ORM names and renamed with as: 
from app.db.models import ExtractedQuestion as ExtractedQuestionORM
# ImportError: cannot import name 'ExtractedQuestion' from 'app.db.models'

10. What is not running and why #

ServiceSkipped becauseImpact
worker-gpu Docker Desktop on this host has no NVIDIA runtime The gpu.heavy Celery queue (embeddings) has no consumer. Ingest → extract works; embedding → clustering → FAQ stalls.
ollama / ollama-init .env points the LLM at Groq, not Ollama None. Dead code in this configuration.

To unblock embeddings

  1. Install NVIDIA Container Toolkit on Docker Desktop and bring worker-gpu up, or
  2. Add a CPU embedding worker: clone Dockerfile.worker-gpu, drop the CUDA base, set EMBEDDING_DEVICE=cpu, listen on the same gpu.heavy queue. Slower but functional.
